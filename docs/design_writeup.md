# VoxGate — Design Writeup

*Post-hoc evaluation for an existing clinic-scheduling voice agent. This is the argument
behind the structure in `eval_system/`; see `README.md` for how to run it.*

## 1. Reproducibility of a non-deterministic, real-time system

A voice agent is non-deterministic on two axes at once: what it *says* (model sampling,
tool-call timing) and *when things happen* (when the caller interrupts, how long a pause
runs). If your test caller is a live bot reacting to the agent in real time, a barge-in
test means a different thing every run — the interruption lands at a different offset,
against a different partial sentence, so "the agent yielded" isn't comparable across runs.

VoxGate is **open-loop, fixed-clock fixture replay**: the caller side of every call is a
**pre-rendered artifact**, not a live participant. A fixture is `call.wav` (2-channel: caller
+ agent), `transcript.jsonl` / `tool_log.jsonl` / `events.jsonl` (authored ground truth on
one canonical clock), plus `scenario_db.json` / `expected.json` (initial DB state + success
criteria). The caller's interruption timing is baked into the caller channel's waveform at
authorship time, not generated by reacting to the agent live — so "the caller barges in at
2.0s into the agent's line" is a property of the fixture, identical on every replay. Nothing
about the caller's behavior depends on the agent's real-time output, which is what makes a
barge-in test at all reusable: score it once, and every future agent version is compared
against the *exact same* interruption.

The one thing this can't give you for free is genuinely concurrent audio: two speakers
talking at once. We hit this directly — a synthetic fixture's "genuine barge-in" scenario
was authored with events claiming an overlap, but the caller's clip was rendered
*sequentially after* the agent's cutoff, not concurrently, so the audio had zero actual
overlap (see `docs/ERRORS.md`, 2026-06-30). The fix wasn't to trust the authored event
timestamp; it was to render the caller clip at an **absolute time inside** the agent's
still-speaking window (`CallBuilder.say_at`). The lesson generalizes: an authored event is a
*claim* about the audio, and a fixture is only reproducible in the sense that matters if
that claim is independently verifiable by inspecting the samples. We don't fabricate
alignment data anywhere in this system for exactly this reason — `MetricContext`'s canonical
clock (audio sample index at a fixed `sr`) is populated from the fixture loader's real
channel/timestamp data, never invented to make a metric's test pass.

Closed-loop (a bot caller that reacts live to the agent) has real value — it's the only way
to find behaviors you didn't think to author — but it buys that at the cost of exactly the
reproducibility this system exists to guarantee. It belongs in an separate, harder-to-CI-gate
exploratory harness, not in the fixed-clock replay that decides ship/no-ship.

## 2. Taxonomy

Two suites, each split by *how* a check is computed, not what it's about:

| Suite | Metric | Kind | Gate? | Why |
|---|---|---|---|---|
| Semantic | `task_success` | deterministic | gate | Final tool call vs. ground-truth success criteria. No proxy involved. |
| Semantic | `tool_call_ordering` | deterministic | gate | State-reducer over tool events, incl. the reschedule-trap invariant. A real correctness bug, not a style call. |
| Semantic | `instruction_adherence` (rule) | deterministic | gate | Substring check that critical entities were read back — objective, ground-truthed. |
| Semantic | `instruction_adherence` (judge) | LLM judge | advisory | Conversational nuance a keyword check can't capture. |
| Semantic | `faithfulness` | LLM judge | advisory→gate | See §3 — the flagship trust-calibration case. |
| Acoustic | `barge_in` | VAD signal | gate | Headline metric; a real miss (fail-to-yield or false-yield) is a real defect once alignment is right. |
| Acoustic | `turn_taking_latency` | VAD signal | advisory | Reports a *distribution* (p50/p90/p99), not a single verdict — advisory by construction. |
| Acoustic | `latency_thresholds` | deterministic | advisory→gate | Timestamp arithmetic against a fixed cutoff; the cutoff itself is a judgment call, so it starts advisory (C1(7)) until the threshold earns trust. |
| Acoustic | `pitch_prosody` | signal | advisory | F0/rate are perceptual proxies for naturalness, not correctness. |
| Acoustic | `entity_intelligibility` | signal (STT+WER) | gate | Round-trip STT on ground-truthed critical entities — if a real STT engine can't recover it, neither could the caller. |
| Acoustic | `emotional_appropriateness` | LLM judge | advisory, always | See §3. |
| Acoustic | `double_talk` | VAD signal | advisory | Overlap isn't inherently bad (backchannels overlap constantly); descriptive, not a verdict. |

Deterministic wins whenever the ground truth is a fact you can assert against code: did tool
X get called with arg Y, does the transcript contain string Z, is a timestamp past a
threshold. Judges earn their place only where the criterion is genuinely a *judgment* — is
this claim actually grounded, is this tone actually appropriate — that no string match can
capture. Signal-processing (VAD/F0/STT) sits in between: mechanically deterministic once you
trust the extractor, but the extractor itself is a model with real failure modes (see §1's
barge-in story, and §3).

## 3. The validity problem

Every acoustic/judge metric here is a proxy, and proxies fail differently, so I gate on
**how independently checkable the criterion is**, not on how sophisticated the extractor
is:

- **`task_success` / `tool_call_ordering` / `instruction_adherence` (rule)** gate immediately:
  the ground truth is a fact in `expected.json`, and the check is a deterministic function of
  it. No calibration needed.
- **`barge_in` / `entity_intelligibility`** gate too, but they run a real model (VAD / STT)
  first. I still call this a real defect if it fires because the failure surface is narrow
  and the model's job is mechanical (was there overlapping voice activity; did this exact
  token survive transcription) — but building this metric surfaced two proxy-validity gaps
  worth being honest about: silero-VAD correctly declined to classify an injected cough as
  speech (a real, useful rejection — not every noise burst is a barge-in), and a naive
  substring match against STT output false-failed on "four eight two one three" vs. STT's
  rendered "4-8213" — a representation mismatch, not an actual intelligibility failure,
  fixed by normalizing spoken-number words to digits before comparing (see
  `entity_intelligibility.py`).
- **`faithfulness`** (LLM judge: are the agent's claims grounded in tool results?) is the
  clearest case for a judge over a rule: "grounded" requires understanding paraphrase and
  implication, not keyword matching. But an LLM judge is itself an unproven evaluator until
  measured, so it starts **advisory**. `calibration/judge_agreement.py` computes Cohen's
  kappa between the judge's verdicts and a small human-labeled set; only once kappa clears a
  threshold (0.6 here — a defensible floor for "beyond chance," not a magic number) does
  `evaluate_gate()`'s `trusted_judge_metrics` argument let it hard-gate. Degenerate,
  no-variability label sets are deliberately never treated as "trusted" (0/0 kappa is
  undefined, not perfect) — see `judge_agreement.py`'s guard.

  **Category-3 note (PHI/BAA):** this agent handles real patient scheduling data — names,
  appointment times, confirmation numbers, and reason-for-visit conversation content are all
  plausibly PHI under HIPAA. Every judge call in this system (`faithfulness`,
  `instruction_adherence_judge`, `emotional_appropriateness`) sends transcript text (and, for
  emotion, a prosody summary) to an external LLM provider. Before this runs on real call data
  rather than synthetic fixtures, that requires (a) a signed BAA with the model provider
  (Anthropic/OpenAI) covering this specific use, (b) minimizing what's sent — the faithfulness
  prompt already only needs tool-result facts + agent statements, not the caller's full turn,
  which reduces incidental PHI exposure, (c) de-identifying or redacting predictable
  high-risk fields (patient name, DOB, MRN if ever present) before judge calls where the
  criterion doesn't actually need them, and (d) audit-logging what was sent to which
  provider, for breach-notification purposes. None of this is implemented — this is a design
  decision documented here, not fake infra bolted onto the judge client.

- **`emotional_appropriateness`** never gates, period, no matter what calibration says
  later — this is a hardcoded invariant, not a default. It's also honestly a *weaker* proxy
  than I'd like: no multimodal-audio-capable judge client is wired up here, so it grades a
  **prosody summary in words** (F0 std/range, a monotone flag, speech rate) plus the
  transcript, not the actual audio. That's a real gap between what it measures and what a
  human perceives, and it's exactly why "always advisory" is the right call rather than
  something calibration could ever promote.
- **Drift**: `calibration/drift.py` KS-tests a judge's current score distribution against a
  frozen golden-set baseline; a significant shift (model update, prompt change,
  provider-side regression) flags for re-calibration before the judge's trust tier is
  trusted going forward.

## 4. Thresholds and gating

`gating/gate.py`'s `evaluate_gate()` fuses one call's `MetricScore` list into a ship/hold
verdict with a rule simple enough to reason about under pressure: **only metrics tagged
GATE, or judges explicitly promoted via a calibration-derived trusted set, can hold a call;
ADVISORY never can, however bad its score.** This is what keeps a flaky judge from failing a
good deploy — an untrusted or advisory judge's FAIL is *information in the report*, never a
blocker. Symmetrically, it's what keeps an unproven judge from silently passing a bad one:
it simply isn't consulted for the verdict at all until it's earned that authority.

"pass^k" here means shipping requires **all** k gate-eligible metrics to pass — a
conjunction, not an average, so one clear violation holds the call regardless of how good
everything else looks. An `ERROR` (the evaluator itself broke — a crash, a timeout, an API
failure) is fail-closed on a gate metric: we don't know if the underlying behavior was fine,
so we don't ship on the strength of a metric that didn't actually run. It's reported
distinctly from `FAIL` so a human can immediately tell "the agent broke a rule" from "our
tooling broke," which matters when triaging a hold. `SKIPPED` (a metric that legitimately
doesn't apply — e.g. `task_success` on a fixture with no `success_criteria.final_tool`) is
excluded from the conjunction entirely rather than counted as a missing pass, so a metric
that doesn't apply to a given call can't accidentally block it.

**Category-2 note (self-consistency sampling):** the more statistically rigorous version of
"a flaky judge shouldn't decide anything alone" is repeating a borderline judge call k times
and requiring self-agreement before trusting a single verdict. That's a real technique, and
a natural extension of `metrics/sampling.py`'s existing stratified-coverage policy — but it's
scoped here as a documented design decision, not implemented, per this system's "no fake
infra" boundary: building a believable self-consistency sampler needs a real budget/latency
tradeoff behind it, not a stub that always agrees with itself.

## 5. Offline vs. online

Fixtures can tell you: does the agent handle *this exact* reschedule trap, *this exact*
cough-vs-barge-in pair, correctly and reproducibly, forever. They cannot tell you what real
callers actually sound like, how often barge-ins happen in practice, whether latency drifts
under real production load, or whether abandonment correlates with anything measurable.
`monitoring/production_proxies.py` runs the `requires_ground_truth = False` subset (VAD/
prosody/judge metrics that don't need `expected.json`) directly against live traffic — the
same evaluators, no separate "prod" implementation to keep in sync. What's worth watching
live that a fixture set structurally can't surface: time-to-yield drift across real caller
population, transfer/abandonment rate correlated with turn-taking latency, and judge-score
drift (`calibration/drift.py`) against the frozen golden set, since a judge that quietly
started scoring differently is exactly the failure fixtures alone won't catch between
scheduled re-calibrations.

**Category-2 note (production infra):** a real broker-backed dead-letter queue for
malformed/failed production recordings, distributed worker pool for scoring at call volume,
and multi-key rotation across judge providers are all real requirements for running this at
production scale — and are deliberately out of scope for this build. Building a *believable*
version of any of them (real retry semantics, real backpressure, real key-rotation failure
modes) is a bigger investment than a stub that doesn't reflect an actual broker/queue's
behavior, and a fake one would be actively misleading in a design review. `validators/
preflight.py`'s retry-then-quarantine (C1(4)) is the one piece of this that *is* built,
because it's genuinely small and testable without fake infra behind it.
